# wje-101 咖啡品鉴社区

## 项目概述

构建一个面向咖啡爱好者的全栈 Web 应用——**咖啡品鉴社区**。用户可以在平台上记录每次咖啡品鉴的详细笔记（包括风味、评分、冲煮方式等），浏览和收藏豆种信息，创建和分享冲煮配方，并通过评论和点赞与其他用户互动。平台还提供个人主页统计功能，展示用户的品鉴历史和偏好画像。

- **项目类型**：全栈 Web 应用（前后端分离 + Docker Compose 一键部署）
- **技术栈**：前端 Vue 3 + TypeScript + Element Plus + Vite，后端 Python + FastAPI + SQLAlchemy，数据库 PostgreSQL
- **COMPOSE_PROJECT_NAME**：`wjecoffeetaste`
- **端口分配**：前端 `28601:80`，后端 `29601:8000`，数据库 `57601:5432`

## 一、硬性约束

### 1.1 构建可运行性

- 所有代码必须可直接运行，禁止出现占位符、TODO 注释、省略号（`...`）、"待实现"等字样
- 前端 `npm run build` 必须零错误通过，TypeScript 类型检查无报错
- 后端 `uvicorn` 启动后所有路由可正常访问，Swagger 文档 (`/docs`) 可正常加载
- 数据库迁移脚本（或自动建表）必须在容器首次启动时完成，包含预置数据
- 所有 API 接口必须返回统一格式的 JSON 响应（包含 code、message、data 字段）

### 1.2 Docker 一键启动

- 项目根目录必须包含 `docker-compose.yml`、`.env.example`、`README.md`
- 执行 `cp .env.example .env && docker compose up -d --build` 后，三个容器（db、backend、frontend）全部健康运行
- 前端容器使用 Nginx 反向代理 `/api` 前缀请求到后端容器
- 数据库容器 healthcheck 通过后后端容器才启动
- 使用命名卷 `db_data` 持久化数据库数据

### 1.3 功能实现完整性

- 每个核心实体的 CRUD 操作必须完整实现，不能只做部分接口
- 每个核心页面必须可正常渲染和交互，不能出现空白页或报错页
- 用户认证流程必须端到端打通：注册 → 登录 → token 刷新 → 权限拦截 → 前端路由守卫
- 品鉴笔记必须支持完整的创建、查看、编辑、删除操作
- 豆种库必须支持管理员维护和用户浏览筛选
- 冲煮配方必须支持创建、关联到品鉴笔记

### 1.4 代码质量

- 前端组件单文件不超过 300 行，逻辑复杂时拆分为组合式函数
- 后端单个函数不超过 50 行，复杂逻辑下沉到 service 层
- 所有 TypeScript 接口必须有完整的字段类型定义，禁止使用 `any`
- 后端 Pydantic schema 必须包含字段校验规则
- 代码中必须有合理的中文注释，关键业务逻辑处必须注释说明意图

## 二、项目主要需求

### 2.1 核心实体（5 个，贯穿全栈）

#### 实体 1：User（用户）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer, PK | 自增主键 |
| username | String(50), unique | 用户名 |
| email | String(100), unique | 邮箱 |
| password_hash | String(128) | 密码哈希（bcrypt） |
| avatar | String(255) | 头像 URL，可为空 |
| bio | String(500) | 个人简介，可为空 |
| role | String(20), default='user' | 角色：user / admin |
| created_at | DateTime | 注册时间 |

**关系**：拥有多个 TastingNote，拥有多个 BrewRecipe，发出多个 Comment，可关注多个 User（自关联多对多：user_followers 表）

#### 实体 2：TastingNote（品鉴笔记）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer, PK | 自增主键 |
| user_id | Integer, FK → User.id | 所属用户 |
| coffee_name | String(100) | 咖啡名称 |
| origin | String(100) | 产地 |
| roast_level | String(20) | 烘焙度：light / medium / dark |
| flavor_tags | JSON | 风味标签数组，如 ["柑橘", "坚果", "巧克力"] |
| aroma_score | Integer | 香气评分 1-10 |
| acidity_score | Integer | 酸度评分 1-10 |
| body_score | Integer | 醇厚度评分 1-10 |
| overall_score | Integer | 综合评分 1-10 |
| brew_method | String(50) | 冲煮方式 |
| brew_recipe_id | Integer, FK → BrewRecipe.id, nullable | 关联冲煮配方 |
| notes_text | Text | 品鉴文字描述 |
| image_url | String(255) | 笔记配图 URL，可为空 |
| created_at | DateTime | 创建时间 |
| updated_at | DateTime | 更新时间 |

**关系**：属于 User，关联 BrewRecipe（可选），拥有多个 Comment，可引用 CoffeeBean

#### 实体 3：BrewRecipe（冲煮配方）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer, PK | 自增主键 |
| user_id | Integer, FK → User.id | 创建者 |
| name | String(100) | 配方名称 |
| device | String(50) | 冲煮器具（如 V60、法压壶、意式咖啡机） |
| water_temp | Float | 水温（℃） |
| grind_size | String(50) | 研磨度（如 细砂糖、中粗） |
| ratio | String(20) | 粉水比（如 1:15） |
| steps | JSON | 冲煮步骤数组，每步含 step_number、description、duration_seconds |
| created_at | DateTime | 创建时间 |

**关系**：属于 User，被多个 TastingNote 引用

#### 实体 4：Comment（评论）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer, PK | 自增主键 |
| note_id | Integer, FK → TastingNote.id | 所属品鉴笔记 |
| user_id | Integer, FK → User.id | 评论者 |
| content | Text | 评论内容 |
| created_at | DateTime | 评论时间 |

**关系**：属于 TastingNote 和 User

#### 实体 5：CoffeeBean（豆种）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer, PK | 自增主键 |
| name | String(100) | 豆种名称 |
| origin | String(100) | 产地 |
| process_method | String(50) | 处理法（水洗、日晒、蜜处理等） |
| flavor_tags | JSON | 风味标签数组 |
| description | Text | 描述 |

**关系**：被多个 TastingNote 引用（逻辑关联，通过 coffee_name 或外键）

#### 关联表：Like（点赞）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer, PK | 自增主键 |
| user_id | Integer, FK → User.id | 点赞者 |
| note_id | Integer, FK → TastingNote.id | 被点赞笔记 |
| created_at | DateTime | 点赞时间 |
| (user_id, note_id) | UniqueConstraint | 每人每笔记仅可点赞一次 |

#### 关联表：UserFollow（关注）

| 字段 | 类型 | 说明 |
|------|------|------|
| follower_id | Integer, FK → User.id | 关注者 |
| following_id | Integer, FK → User.id | 被关注者 |
| created_at | DateTime | 关注时间 |
| (follower_id, following_id) | PK | 复合主键 |

### 2.2 核心页面（6 个）

#### 页面 1：首页 `/`

- **消费实体**：TastingNote + User + Like
- **功能**：信息流展示最新品鉴笔记（分页），热门笔记排行（按点赞数排序），支持按烘焙度、产地筛选
- **布局**：顶部导航栏（Logo、搜索框、用户头像/登录按钮），主体分左右两栏——左侧为信息流卡片列表，右侧为热门排行和推荐豆种

#### 页面 2：创建品鉴笔记 `/note/create`

- **消费实体**：User + CoffeeBean + BrewRecipe
- **功能**：表单创建品鉴笔记，填写咖啡名称（可从豆种库选择自动填充）、产地、烘焙度（下拉选择枚举值）、风味标签（多选标签组）、各项评分（1-10 滑块或星级选择）、冲煮方式、关联冲煮配方（下拉选择已有配方）、品鉴文字描述、上传配图
- **布局**：居中表单布局，左右分栏——左侧为基本信息和评分，右侧为冲煮配方关联和详细描述

#### 页面 3：品鉴详情 `/note/:id`

- **消费实体**：TastingNote + Comment + User + Like + BrewRecipe
- **功能**：展示完整品鉴信息（作者头像和用户名、咖啡信息、评分雷达图或星级展示、风味标签组、文字描述、配图），关联冲煮配方卡片，评论区（发表评论、展示评论列表），点赞按钮和计数，编辑/删除按钮（仅作者可见）
- **布局**：上部为笔记主内容，下部为评论区

#### 页面 4：豆种库 `/beans`

- **消费实体**：CoffeeBean
- **功能**：浏览全部豆种卡片，按产地筛选、按处理法筛选、按风味标签筛选、关键词搜索，管理员可新增/编辑/删除豆种
- **布局**：顶部筛选栏 + 下方网格卡片布局

#### 页面 5：个人主页 `/profile/:id`

- **消费实体**：User + TastingNote + UserFollow
- **功能**：展示用户头像、用户名、简介、关注/粉丝数、品鉴统计（品鉴次数、平均分、最爱产地 TOP3），品鉴历史列表，关注/取关按钮
- **布局**：上部用户信息卡片，中部统计面板（数字 + 图表），下部品鉴笔记列表

#### 页面 6：配方广场 `/recipes`

- **消费实体**：BrewRecipe + User
- **功能**：浏览所有公开冲煮配方，按器具类型筛选，按水温范围筛选，搜索配方名称，点击可查看配方详情（完整步骤）
- **布局**：顶部筛选栏 + 下方卡片列表

### 2.5 共享前端组件（4 个）

#### 组件 1：`<ScoreStars>`

- **路径**：`frontend/src/components/common/ScoreStars.vue`
- **功能**：接收 `score`（1-10）和 `max`（默认 10）props，以半星形式展示评分，支持 `readonly` 和可交互两种模式
- **消费页面**：品鉴详情页、品鉴列表页卡片、个人主页统计区

#### 组件 2：`<FlavorTags>`

- **路径**：`frontend/src/components/common/FlavorTags.vue`
- **功能**：接收 `tags`（字符串数组）和 `editable`（布尔）props，以彩色标签组形式展示风味标签，editable 模式下支持添加和删除标签
- **消费页面**：品鉴创建页（可编辑模式）、品鉴详情页（只读模式）、豆种详情页（只读模式）

#### 组件 3：`<EmptyState>`

- **路径**：`frontend/src/components/common/EmptyState.vue`
- **功能**：接收 `icon`、`title`、`description` props，展示空状态插图和提示文字
- **消费页面**：品鉴列表（暂无笔记）、配方列表（暂无配方）、关注列表（暂无关注）

#### 组件 4：`<UserAvatar>`

- **路径**：`frontend/src/components/common/UserAvatar.vue`
- **功能**：接收 `user` 对象和 `size`（small / medium / large）props，展示用户头像（有头像显示图片，无头像显示用户名首字），点击跳转个人主页
- **消费页面**：顶部导航栏、评论区每条评论、关注列表

### 2.6 共享 Hooks（2 个）

#### Hook 1：`useAuth`

- **路径**：`frontend/src/hooks/useAuth.ts`
- **功能**：
  - `login(email, password)` — 调用登录接口，存储 token 到 localStorage
  - `register(username, email, password)` — 调用注册接口
  - `logout()` — 清除 token，跳转首页
  - `currentUser` — 响应式当前用户信息
  - `isLoggedIn` — 响应式登录状态
  - `isAdmin` — 是否管理员
  - `refreshToken()` — 刷新 token

#### Hook 2：`usePagination`

- **路径**：`frontend/src/hooks/usePagination.ts`
- **功能**：
  - 封装通用分页逻辑：`page`、`pageSize`、`total`、`loading` 状态
  - `fetchPage(apiFn, params)` — 调用分页接口并更新状态
  - `goNext()` / `goPrev()` — 翻页
  - 返回响应式状态供模板绑定

### 2.7 后端中间件（3 个）

#### 中间件 1：`auth_middleware`

- **路径**：`backend/app/middlewares/auth.py`
- **功能**：从请求头 `Authorization: Bearer <token>` 解析 JWT，提取 user_id 和 role，注入到 request.state。未携带 token 或 token 过期返回 401。提供 `require_auth` 和 `require_admin` 两个装饰器供路由使用。

#### 中间件 2：`error_handler_middleware`

- **路径**：`backend/app/middlewares/error_handler.py`
- **功能**：注册为 FastAPI 全局异常处理器。捕获 `HTTPException` 和未预期的 `Exception`，统一返回 `{ "code": int, "message": str, "data": null }` 格式。记录异常堆栈到日志。

#### 中间件 3：`logger_middleware`

- **路径**：`backend/app/middlewares/logger.py`
- **功能**：记录每个 HTTP 请求的方法、URL、查询参数、响应状态码、处理耗时（毫秒）。使用 Python logging 模块，输出格式包含时间戳。

## 四、核心流程

### 流程 1：用户注册并登录

1. 用户访问任意页面 → 前端路由守卫检测未登录 → 跳转 `/login`
2. 用户填写用户名、邮箱、密码 → 前端调用 `POST /api/auth/register`
3. 后端 auth 路由接收请求 → user_service 创建用户（bcrypt 哈希密码）→ 返回 success
4. 前端自动调用 `POST /api/auth/login`（或提示用户登录）
5. 后端验证邮箱密码 → 生成 JWT（含 user_id, role, 过期时间）→ 返回 token
6. 前端 `useAuth` hook 将 token 存入 localStorage → 更新 currentUser 状态
7. 前端 axios 拦截器后续每次请求自动附加 `Authorization: Bearer <token>`

### 流程 2：创建品鉴笔记

1. 用户点击导航栏"写笔记"按钮 → 跳转 `/note/create`
2. 页面加载时：并行请求豆种列表 `GET /api/beans`、用户配方列表 `GET /api/recipes?user_id=me`
3. 用户填写表单：选择或手动输入咖啡名称、选择产地、选择烘焙度（RoastLevel 枚举下拉）、添加风味标签（FlavorTags 组件，可选预设标签或自定义）、拖动评分滑块（1-10）、选择关联冲煮配方（可选）、填写品鉴描述
4. 点击提交 → 前端校验必填字段 → 调用 `POST /api/notes`
5. 后端 note 路由 → note_controller → note_service：创建 TastingNote 记录 → 返回新笔记 ID
6. 前端收到成功响应 → 跳转到 `/note/:id` 展示新建笔记

### 流程 3：浏览品鉴详情并互动

1. 用户点击首页笔记卡片 → 跳转 `/note/:id`
2. 前端并行请求：`GET /api/notes/:id`（笔记详情 + 作者信息）、`GET /api/notes/:id/comments`（评论列表）
3. 页面渲染：UserAvatar 展示作者、ScoreStars 展示评分、FlavorTags 展示风味标签、关联配方卡片
4. 用户点击点赞按钮 → 调用 `POST /api/notes/:id/like` → 后端 Like 表插入记录 → 前端更新点赞数
5. 用户在评论区输入文字 → 调用 `POST /api/notes/:id/comments` → 后端 Comment 表插入记录 → 前端刷新评论列表
6. 仅笔记作者本人看到编辑/删除按钮 → 删除调用 `DELETE /api/notes/:id`

### 流程 4：个人主页统计

1. 用户访问 `/profile/:id` → 前端请求 `GET /api/users/:id`、`GET /api/notes?user_id=:id`
2. 后端 user_service 计算统计：品鉴次数（COUNT notes）、平均分（AVG overall_score）、最爱产地（GROUP BY origin ORDER BY COUNT DESC LIMIT 3）
3. 前端 Profile.vue 渲染：用户信息卡片 + 统计数字 + ECharts 雷达图（香气/酸度/醇厚度平均分） + 品鉴历史列表
4. 其他用户可点击关注按钮 → 调用 `POST /api/users/:id/follow` → 更新 UserFollow 表

### 流程 5：豆种库浏览与筛选

1. 用户访问 `/beans` → 前端请求 `GET /api/beans`
2. 豆种列表以网格卡片展示，每张卡片显示名称、产地、处理法、风味标签
3. 用户选择筛选条件（产地下拉、处理法下拉、风味标签多选）→ 前端带参数请求 `GET /api/beans?origin=xxx&process=xxx&flavor=xxx`
4. 后端 bean_service 构建动态查询 → 返回筛选结果
5. 管理员用户可见"新增豆种"按钮 → 弹出表单对话框 → 提交 `POST /api/beans`

## 六、代码存放规则

1. **所有代码必须位于 `/workspace` 目录下**（Docker 容器内对应此路径，宿主机对应项目根目录）
2. 项目根目录结构：
   ```
   /workspace/
   ├── frontend/          # 前端项目（Vue 3 + Vite）
   ├── backend/           # 后端项目（FastAPI）
   ├── docker-compose.yml
   ├── .env.example
   └── README.md
   ```
3. 前端和后端分别有自己的 `Dockerfile`
4. 前端 `nginx.conf` 放置在 `frontend/nginx.conf`
5. 后端 `requirements.txt` 放置在 `backend/requirements.txt`
6. **严禁**将后端代码放在前端目录内，反之亦然
7. **严禁**在根目录直接放置源代码文件（只允许 docker-compose.yml、.env.example、README.md）

## 七、部署要求

### 7.1 docker-compose.yml 骨架（PostgreSQL 版）

以下为必须严格遵循的 Docker Compose 配置。不写 `version` 字段，使用顶层 `name` 字段。

```yaml
name: wjecoffeetaste

services:
  db:
    image: postgres:16-alpine
    container_name: ${COMPOSE_PROJECT_NAME:-wjecoffeetaste}-db
    restart: unless-stopped
    environment:
      POSTGRES_DB: ${DB_NAME}
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes:
      - db_data:/var/lib/postgresql/data
    ports:
      - "${DB_PORT:-57601}:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER} -d ${DB_NAME}"]
      interval: 5s
      timeout: 5s
      retries: 20

  backend:
    build: ./backend
    container_name: ${COMPOSE_PROJECT_NAME:-wjecoffeetaste}-backend
    restart: unless-stopped
    environment:
      DATABASE_URL: postgresql+asyncpg://${DB_USER}:${DB_PASSWORD}@db:5432/${DB_NAME}
      JWT_SECRET: ${JWT_SECRET}
    ports:
      - "${BACKEND_PORT:-29601}:8000"
    depends_on:
      db:
        condition: service_healthy

  frontend:
    build: ./frontend
    container_name: ${COMPOSE_PROJECT_NAME:-wjecoffeetaste}-frontend
    restart: unless-stopped
    ports:
      - "${FRONTEND_PORT:-28601}:80"
    depends_on:
      - backend

volumes:
  db_data:
```

### 7.2 .env.example

```env
COMPOSE_PROJECT_NAME=wjecoffeetaste
DB_NAME=coffeetaste
DB_USER=coffeetaste
DB_PASSWORD=coffeetaste_pwd
DB_PORT=57601
JWT_SECRET=change_me_to_a_long_random_string
FRONTEND_PORT=28601
BACKEND_PORT=29601
```

### 7.3 前端 Dockerfile 要求

- 使用多阶段构建：第一阶段 `node:20-alpine` 执行 `npm install && npm run build`，第二阶段 `nginx:alpine` 拷贝构建产物
- 将 `frontend/nginx.conf` 拷贝到 Nginx 配置目录
- nginx.conf 必须配置：
  - `location /api/` 反向代理到 `http://backend:8000/`
  - SPA fallback：`try_files $uri $uri/ /index.html`
  - gzip 压缩

### 7.4 后端 Dockerfile 要求

- 基于 `python:3.11-slim`
- 安装 `requirements.txt` 中的依赖
- 启动命令：`uvicorn app.main:app --host 0.0.0.0 --port 8000`
- 应用启动事件中自动执行建表和预置数据插入（幂等操作，已存在则跳过）

### 7.5 数据库注意事项

- 使用 PostgreSQL 16 Alpine 镜像
- 健康检查使用 `pg_isready -U ${DB_USER} -d ${DB_NAME}`
- 使用命名卷 `db_data` 持久化数据
- 后端使用 `postgresql+asyncpg` 驱动连接
- 连接串格式：`postgresql+asyncpg://user:password@db:5432/dbname`
